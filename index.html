Version Control System

¿What is a Version Control System?
A VCS is a tool that helps manage changes in the source code over time. This sytem is responsible for monitoring the code and informs if it has been modified inder to register such change

There are different types of Version Control System, there are:

  • Locals: In this type el repository lives in your computer
  • CentralizedThere is a central repository (a super server) where all the codes is located
  • DistributedEach programmer has his own copy in his computer of the original repository, so the changes you make locally will no affect the rest of the rest of the programmer who are working for the project - GIT is Distributed

¿What is GIT?

Git is a Version Control System, in fact is the most modern and used in the world. Git was developed by Linux Torvalds in 2005, is a open soruce project constantly maintained by the community.

Three states of Git:
The most important thing to know about Git are its three main states:


  • Workin Directory: It is the where the files are worked.
  • Staging area: It is the area of preparation of the files that will be registered in the repository.
  • Git repositoryIt is the place where the code is stored securely.

Git Installation and Configuration


Install GIT on linux:
- apt-get install git

Show the version of GIT installed:
- git --version

Establish an email:
- git config --global user.email [e@e.com]

Set a username:
- git config --global user.name [username]

Enable colors in the UI:
- git config --global color.ui [true or false]

To install GIT on Linux specifically on Debian/Ubuntu or derived distributions, the command is executed:

  • apt-get install git
  • git -- version
  • git config --global user.email [email@email.com]
  • git config --global user.name [username]
  • git config --global
  • git config --global color.ui [true or false]

You can get more informaciòn about strong GIT settings in the Official Git Documentation

 - git diff [commit HASH]

Creating and deleting repositories (git init)

- Create a repository, the name of the folder is optional, in case of placing it said directory will be created, in case of not doing so, the repository will be created starting from the current folder.
git init [dir-name]

- Deleting the .git folder is the same as deleting the repository (locally).
rm -rf .git

Adding, removing and viewing file status (git add | rm | status )

  • git add -A=>Add all the files in the Working directory to the Stanging area
  • git add [file or directory] => Add a file or folder from the Working Directory to the Staging Area
  • git rm --cached [file or directory] => Delete a file or folder the Stanging area and leave it in the Working directory
  • git rm -f [file or directory] => It force the deletion of a file or folder from the Stangin area so much that it even deletes it form the Working directory
  • git status => Show the status of files or directories
    • File that appear in red : They are in the Working Directory.
    File that appear in green: They ara in the Staging Area.

Confirming changes (git commit)

  • git commit -m "[Description]"=> Add the stanging area files to the Git repository with a description
  • git commit -m --amend=> If you make a mistake (commit) with the flag --amend you can rectify and concatenate the changes to the last commit. If the -m "[description ...]" flag is used, the new message will overwrite the previous one, in case of not doing so the old one will be maintained.

Tagging confirmations (git tag)

  • git tag -a [1.x] -m "[description]"=> Register a tag
  • git tag -l=> List exiting tag
  • git tag [1.x] [Commit HASH] =>It registers a tag to a specific commit through its HASH.
  • git tag -f -a [1.x] -m "[description]" [Commit HASH] =>Rename a tag
  • git tag -d [1.x] =>Delete a tag

Reviewing the history of our project (git log)

  • git log => It shows the history of the records (commits)

Reviewing changes between versions (git diff)

  • git diff [commit HASH]It shows the changes that have been made between the current status and the commit specified by your HASH.
  • git diff [commit HASH #1] [commit HASH #2]

When the comparisons are in red, it means that things were removed instead if it green, it means that things were add

Revert changes in Git

- Starting from a Commit identified by its HASH, it eliminates all future Commits to it and keeps all those changes (registered in the deleted Commits) in the Staging Area.
git reset --soft [HASH]

- Starting from a Commit identified by its HASH, it eliminates all future Commits to it and keeps all those changes (registered in the deleted Commits) in the Working Directory.
git reset --mixed [HASH]

- Starting from a Commit identified by its HASH, it eliminates all future Commits to it and does not keep the changes either in the Staging Area or in the Working Directory of the deleted Commits.
git reset --hard [HASH]

Multiple variants of the repository (git branch)
A branch is an alternate timeline (of the project). The default branch is master, the GitHub Pages branch is gh-pages and when working as a collaborator the main project is identified with the branch upstream.

  • git branch [name-branch] => Create a new branch
  • git checkout -b [name-branch] => Create a new branch and position yourself in it
  • git branch -l => List all the branches that exist
  • git branch -d [name-branch] => Delete a branch, with -D deletion force the delete
  • git branch -m [old-name-branch] [new-name-branch] => It allows to rename a new branch

Moving between branches and versions (git checkout)
-This allows moving between branches or between commits of a branch:
git checkout [name-branch or commit-HASH
